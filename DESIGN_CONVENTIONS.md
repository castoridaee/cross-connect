# Cross Connect Design Conventions

This document outlines the design conventions for the Cross Connect game. The game is a puzzle game similar to Connections but with a x-y grid and crossing categories similar to a Crossword puzzle.

## Core Principles

### Gameplay

- **Tutorialization**: Avoid explicit tutorials. Use a fixed puzzle or possibly sequence of puzzles for new users to demonstrate the mechanics through play. Use brief popover boxes to explain mechanics when the user interacts with the puzzle in a way that isn't correct.
- **Puzzle generation**: Puzzles should be user generated, and should be obvious in the UI. The puzzle creator should be simple and obvious, with clear ways to fix mistakes.
- **Puzzle statistics**: Puzzle statistics should be generated by coalesced user behavior. There should be no manual difficulty grading or quality rating (other than the like button). 
- **Puzzle queue**: The primary goal of the puzzle queue is to serve apropriate puzzles for the user based on their skill level, difficulty preference, and the puzzle quality. The secondary goal of the puzzle queue is to make sure that every puzzle gets played ~5 times to establish statistics. Puzzle queue can have some minor easter eggs (for example, always serving a certain puzzle on holidays).

### User Accounts

- **No pressure to sign up**: Users should be able to play without an account. Account signup should be encouraged only after a new puzzle is saved, in order to help them claim ownership of their puzzle.
- **Account signup conversion**: When a user signs up or logs in, it should merge the anonymous account with the user's account, and transfer any puzzles they have created under the anonymous user. In cases where there's possible conflict, the ownership conversion should always favor the user's account (for example, if the anonymous account has solved a puzzle that the logged-in user has already solved, it should prefer the logged in user's record and not merge the anonymous user record).

### Mature Presentation

- **Avoid "cute" or "playful" design elements**: Should evoke a NY Times puzzle aesthetic. It should feel clean and timeless.
- **Fonts**: Use simple Sans Serif fonts.
- **Titles and Headers**: Headers can be all caps, but in cases where the the text is showing a user-submitted title or clue, it should respect the capitalization of the user submission. The only exception is that the text on the text on the tiles should be all-caps.
- **Use of color**: Primary interface should be black and white, with color used only to convey information and direct attention.
- **Use of icons**: When icons are used, they should be simple, clean, and consistent. No color icons, except as to convey active status.
- **Avoid engagement bait**: Avoid using manipulative design patterns. This includes language that's overly enthusiastic or suggesting the user do something.

### Information Density

Prioritize information density over excessive whitespace. The interface should feel compact, efficient, and direct.

**Layout & Spacing**
- **Hierarchy**: Avoid excessive hierarchy. Hierarchy can still be used to convey relationships between elements, but only if there's a clear function of the hierarchy.
- **Paddings**: Avoid large paddings. Prefer `p-2` or `p-3` for containers rather than `p-6` or higher.
- **Gaps**: Use tighter gaps between related elements. Use `gap-2` or `gap-3` where `gap-6` or `gap-8` might be standard in other apps.
- **Margins**: Keep vertical margins minimal between sections.
- **Header**: The main navigation should be compact. Horizontal padding (`px-3` or `px-4`) is preferred over wide margins.

**Typography**
- **Scalability**: Font sizes should be readable but not oversized.
- **Tracking**: Use tight tracking for headings (`tracking-tighter` or `tracking-tight`) to maintain high-density visual flow.

**Interaction**
- **Micro-animations**: Transitions should only be used as an affordance to help the user understand state changes, not for decoration.
- **Compact Components**: Cards and modals should contain information without unnecessary space.

### Encourage a Good Community with Minimal Moderation

The goal of these principles is to maintain a positive community and not feed trolls, while keeping moderation overhead low.

- **Shadowban**: Contentious, insulting, or innapropriate content should be shadowblocked, and the user should never be informed of this directly. The reason is to not alert users and encourage them to be hostile, or to workaround content filters. Shadowblocked comments or puzzles should not show up in feeds or search results, but should still allow static linking, sharing, and direct access or access through the users profile page.
- **Simple automated moderation**: Use low-overhead automated moderation; simple libraries and regex filters. Be smart about things to try and catch >90% of toxic content.
- **No downvotes**: There should be no downvotes or affordances to express negativity.
- **No reporting**: There should be no reporting. I don't want to deal with the moderation overhead.
- **Content**: Anatomy should not be shadowblocked. Severe profanity should be shadowblocked. Minor profanity should be allowed, unless it is directed in a hostile way at a user. Racist/sexist content should be shadowblocked.
- **Usernames**: Offensive usernames should report back as being unavailable in the exact same way that already-used usernames are reported back.

## Design targets

- **Browsers**: Supports latest versions of Chrome, Firefox, and Safari.
- **Sizes**: Supports all features on displays down to iPhone SE (2nd Gen) size (CSS pixels 375x667), and up to desktop displays.
- **Platforms**: Main goal is webapp, but if native apps are made, they should be based on the webapp with minimal platform specific changes.
